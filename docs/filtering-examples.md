# Filtering examples

A single worked tree run through every filter option, with the **actual** sitemap
output each config produces. The outputs below were generated by the shipped
`walk_directory`, not hand-written.

## The model in one paragraph

A filter has two sides. The **include / allowlist** side (`keep`, `keep_file`,
`nested_keep_filename`) decides what is *eligible*; the **ignore / deny** side
(`ignore`, `ignore_file`, `nested_ignore_filename`) then *subtracts* from it. A path
is indexed iff the include side keeps it **and** the ignore side does not drop it -
so **ignore wins** across the two. Within each side the three sources apply
inline -> file -> nested and **later wins** (a deeper nested file beats a shallower
one). With no `keep*` source set, everything is eligible (plain deny mode).

Two things to remember reading the outputs:

- hugesitemap emits a **directory URL** (trailing `/`) for every directory it walks,
  plus a **file URL** per surviving file. `keep` filters *files*; a directory is
  still listed if the walk descends into it (and pruned if nothing under it can
  match).
- Patterns use git `.gitignore` syntax, anchored at each `[[site.directory]]` root.

## The example tree

On-disk content, mapped to the site root `https://example.com/`:

```
content/
|-- index.html
|-- notes.txt
|-- guide/
|   |-- intro.html
|   |-- intro.pdf
|   `-- archive/
|       `-- old.pdf
|-- data/
|   |-- report.pdf
|   `-- report.tmp
`-- cache/
    `-- tmp.pdf
```

```toml
[[site]]
name        = "example"
base_url    = "https://example.com/"
output_path = "/srv/www/sitemap.xml"
  [[site.directory]]
  path = "/srv/www/content"
  url  = "https://example.com/"
  # [site.filters] varies per scenario below
```

In the outputs, `/` is the site-root directory URL, a trailing-slash entry like
`guide/` is a directory URL, and `guide/intro.pdf` is a file URL (each is served as
`https://example.com/<path>`).

## 1. No filter (plain deny mode)

```toml
[site.filters]
# nothing set
```

```
/
cache/
cache/tmp.pdf
data/
data/report.pdf
data/report.tmp
guide/
guide/archive/
guide/archive/old.pdf
guide/intro.html
guide/intro.pdf
index.html
notes.txt
```

Everything: 5 directory URLs + 8 file URLs.

---

## 2. exclude - inline `ignore`

```toml
[site.filters]
ignore = ["*.tmp", "*.txt", "cache/"]
```

```
/
data/
data/report.pdf
guide/
guide/archive/
guide/archive/old.pdf
guide/intro.html
guide/intro.pdf
index.html
```

`notes.txt` and `report.tmp` are dropped; the trailing slash on `cache/` prunes the
whole `cache/` subtree (the directory URL **and** `tmp.pdf`).

---

## 3. include - inline `keep` (allowlist by file type)

```toml
[site.filters]
keep = ["*.pdf"]
```

```
/
cache/
cache/tmp.pdf
data/
data/report.pdf
guide/
guide/archive/
guide/archive/old.pdf
guide/intro.pdf
```

Only PDF **files** survive (`index.html`, `intro.html`, `notes.txt`, `report.tmp`
are gone). Directory URLs all remain: `*.pdf` is unanchored, so any directory could
hold a match and the walk descends into all of them.

---

## 4. include - `keep` a subtree (anchored allowlist)

```toml
[site.filters]
keep = ["guide/**"]
```

```
/
guide/
guide/archive/
guide/archive/old.pdf
guide/intro.html
guide/intro.pdf
```

Only the `guide/` subtree is indexed. `data/` and `cache/` are **pruned** entirely -
nothing under them can match the anchored pattern - so they cost no walk and produce
no URLs. This is the scalable way to narrow a large tree.

---

## 5. nested exclude - `.sitemapignore` per directory

```toml
[site.filters]
nested_ignore_filename = ".sitemapignore"
```

with a file `data/.sitemapignore`:

```
*.tmp
.*
```

```
/
cache/
cache/tmp.pdf
data/
data/report.pdf
guide/
guide/archive/
guide/archive/old.pdf
guide/intro.html
guide/intro.pdf
index.html
notes.txt
```

The rules live next to the content they govern: `data/.sitemapignore` drops
`data/report.tmp` (its `*.tmp` rule) and hides itself (its `.*` rule), and applies
*only* under `data/`. Everywhere else is untouched. The whole tree is scanned once
for these files at startup, so the discovery is eager (a file in a subtree that would
otherwise be pruned is still found).

---

## 6. nested include - `.sitemapinclude` per directory

```toml
[site.filters]
nested_keep_filename = ".sitemapinclude"
```

with a file `guide/.sitemapinclude`:

```
*.html
```

```
/
guide/
guide/archive/
guide/intro.html
```

Setting any include source switches the site to **allowlist mode**. The only keep
rule lives under `guide/` (`*.html`), so only `guide/intro.html` is eligible. `data/`
and `cache/` have no possible match and are pruned; `guide/archive/` is still listed
because an `*.html` *could* live under it (no file does, so it is an empty directory
URL). The site root `/` is always emitted. The `.sitemapinclude` file is not indexed
(it does not match `*.html`).

---

## 7. combine include + exclude (`keep` then `ignore` subtracts)

```toml
[site.filters]
keep   = ["*.pdf"]            # eligible: PDFs everywhere
ignore = ["cache/", "*.tmp"]  # subtract: the cache subtree
```

```
/
data/
data/report.pdf
guide/
guide/archive/
guide/archive/old.pdf
guide/intro.pdf
```

Start from "only PDFs" (scenario 3), then the ignore side removes the `cache/`
subtree - so `cache/tmp.pdf` is gone even though it is a PDF. This is the headline
combination: **`ignore` always wins over `keep`**. You allowlist broadly, then carve
out exceptions.

---

## Precedence, restated

1. **Include phase.** If any `keep*` source is set, a path must be kept by it to be
   eligible; otherwise everything is eligible.
2. **Ignore phase.** Among the eligible, any `ignore*` match drops the path.
3. **Within a side**, sources apply inline -> rule file -> nested files, later
   winning, and a deeper nested file beats a shallower one.
4. **Across the sides**, ignore wins: `indexed = kept AND NOT ignored`.

See [CONFIG.md](../CONFIG.md#filters-gitignore-semantics) for the field reference and
the global-vs-site merge rules.
